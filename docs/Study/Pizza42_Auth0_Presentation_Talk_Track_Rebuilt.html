Pizza42 + Auth0 Presentation Talk Track

GoalSet the tone and explain the business goal.
Suggested Talk TrackToday I'll walk through how Pizza42 could modernize customer identity using Auth0. The goal wasn't simply implementing login — it was designing a secure and scalable customer identity architecture.

Slide 2 — Business Goals

GoalShow empathy for different stakeholder priorities.
Suggested Talk TrackSecurity wanted reduced credential liability. Product wanted frictionless onboarding. Marketing wanted richer customer identity data. Engineering wanted scalable OAuth security.
Presentation TipFocus on business value, not protocols.

Slide 3 — Why Auth0

GoalExplain why Auth0 solves these challenges well.
Suggested Talk TrackAuth0 centralizes authentication, secures APIs, reduces password liability, and allows Pizza42 to personalize customer experiences using modern CIAM patterns.
Presentation TipAvoid deep technical detail here.

GoalDemonstrate every requirement was implemented.
Suggested Talk TrackThis checklist guided the architecture and implementation decisions throughout the build.

Live Demo

GoalShow the solution working.
Suggested Talk TrackWalk through signup, login, blocked unverified order, verified order, metadata update, and personalized order history.

Slide 6 — Identity Responsibilities

GoalExplain separation of responsibilities.
Suggested Talk TrackAuthentication, authorization, and privileged backend operations were intentionally isolated into separate trust boundaries.
Presentation TipKey architecture slide.

Slide 7 — OAuth Flows

GoalExplain OAuth choices.
Suggested Talk TrackThe SPA uses Authorization Code Flow with PKCE. The backend uses Client Credentials. Refresh Token Rotation improves long-lived session security.
Presentation TipEmphasize intentional design decisions.

Slide 8 — Tokens

GoalClarify token roles.
Suggested Talk TrackThe ID token represents user identity. The access token authorizes APIs. The refresh token maintains long-lived sessions.
Presentation TipKeep this simple.

Slide 9 — Scopes vs RBAC

GoalShow scalability thinking.
Suggested Talk TrackThe POC intentionally starts lightweight with scopes, while still allowing the platform to evolve naturally toward RBAC later.
Presentation TipDemonstrates architectural maturity.

Slide 10 — Passkeys

GoalConnect passkeys to stakeholder value.
Suggested Talk TrackPasskeys improve both security and customer experience simultaneously while reducing password-reset burden.
Presentation TipTalk business outcomes.

Slide 11 — Tenant Configuration

GoalShow implementation credibility.
Suggested Talk TrackThis slide shows the tenant-side implementation supporting the demo: applications, APIs, Actions, scopes, and security controls.
Presentation TipKeep concise.

Slide 12 — Code Walkthrough

GoalExplain implementation logic.
Suggested Talk TrackThe backend validates every access token independently, while Actions enrich tokens with personalized claims.
Presentation TipAvoid deep code review energy.

Slide 13 — Security Controls

GoalExplain defense in depth.
Suggested Talk TrackNo single control is trusted alone. The architecture layers JWT validation, scope enforcement, verification, rotation, and secrets isolation.
Presentation TipStrong security credibility slide.

Slide 14 — Key Takeaways

GoalReconnect to business value.
Suggested Talk TrackThe result is a scalable CIAM architecture balancing security, customer experience, and operational simplicity.
