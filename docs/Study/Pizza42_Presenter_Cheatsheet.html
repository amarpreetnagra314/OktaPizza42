STEP 3
~3 min

Verify email — focus listener unlocks the UI without refresh

This is your proudest engineering moment. Most demos require a logout/login cycle here. Yours doesn't. Sell it.
Show
1. Switch to the email tab. Click the Auth0 verification link.
2. Auth0 confirmation page appears.
3. Switch back to the Pizza 42 tab. Don't click anything yet.
4. Watch the banner disappear and the order button enable — automatically.
5. Decode the new token in jwt.io → email_verified: true.
Say
Watch the tab — I haven't clicked anything. When the window regained focus, a focus listener I wired into the SPA called getAccessTokenSilently with cache mode off. That forced the SDK to send a back-channel POST to Auth0's token endpoint using the refresh token. Auth0 re-ran the Post-Login Action, issued a fresh ID token with the current email_verified value, the SDK swapped the cached token, and React re-rendered. The user goes from inbox to verified to able-to-order without any manual refresh. That's the kind of polish that separates a working PoC from a real product.
If they ask why refresh tokens, not iframe silent renew
Iframe renewal depends on the Auth0 SSO cookie flowing as a third-party cookie. Chrome, Safari, and Brave increasingly block that. Refresh tokens use a back-channel POST — no cookies — and work everywhere. I enabled them with rotation: each refresh issues a new token and revokes the old one, with reuse detection as a breach signal.
STEP 4
~3 min

Place an order — three gates, one HTTP call

Show
1. Pick "Margherita" + price. Click Place Order.
2. Switch to Vercel logs (or DevTools network) — show the backend's verification: signature ✓, scope ✓, email_verified ✓.
3. Show the Mgmt API PATCH to app_metadata.orders.
4. Order success message appears in the UI.
Say
One click in the UI; three independent gates on the backend. Gate one — JWT signature, audience, issuer all verified against Auth0's JWKS. Gate two — the access token's scope claim must contain create:orders. That's authorization, separate from authentication. Gate three — the backend calls Auth0's Management API live to check email_verified. That's not from the token, which could be a stale snapshot. It's the current state from Auth0 itself. Only when all three pass does the backend write the order — to app_metadata, not user_metadata, because orders are system-owned and tamper-proof.
If they ask why a live Mgmt API call instead of trusting the token
The token's email_verified is a snapshot from login time. A user who verifies mid-session would be blocked until they re-login. The live lookup costs one extra HTTP call — which the backend was making anyway to save the order — and gives always-current state.

Slide 8 — the three-application architecture and why it matters.
8
Separating Identity Responsibilities
~1:30

Slow down here. This is one of the strongest architecture slides and the panel will linger.

Three Auth0 applications, three trust boundaries. The SPA is a public client — anyone can read its JavaScript, so it cannot hold a secret. It uses Authorization Code with PKCE, the only safe choice. The backend API is a resource server — it validates tokens and enforces scope. And the M2M application is a confidential client — it has its own credentials, lives only on the server, and exists for one reason: to call Auth0's Management API with least privilege.
"Why not one app that does everything?"
Three reasons. Different grants — Auth0 doesn't let one app record use both Authorization Code and Client Credentials. Different trust levels — the M2M holds powerful Management API scopes that should never touch a browser. And audit clarity — the Auth0 logs tag every action by app, and mixing them would muddy investigations.
Slide 9 — the OAuth flows and what I deliberately did not choose.
9
OAuth Flows We Selected
~1:30

The "intentionally rejected" panel is what the panel grades you on — it proves reasoning, not copy-paste.

Three flows in this architecture. Authorization Code with PKCE for the SPA — current OAuth 2.1 best practice for browser apps. Client Credentials for the backend M2M — the only correct choice when no user is involved. And refresh tokens with rotation for silent session renewal.
Three flows I deliberately rejected. Implicit — deprecated, puts tokens in the URL. ROPC — requires the app to handle the password, which defeats the whole point of choosing Auth0. And Authorization Code without PKCE — fine for confidential clients but unsafe for SPAs that can't keep a secret.
Do not turn this into an RFC lecture. Three flows in, three flows out, twenty seconds each. The reasoning is what scores, not the depth.
Slide 10 — what each token actually does.
10
Tokens in the Architecture
~0:45

The classic trap: which token does the API trust? Be ready.

Three tokens, three jobs, three audiences — that's the headline. The ID token tells the SPA who the user is. The access token tells the backend what the user is allowed to do. The refresh token lives only between the SPA and Auth0's token endpoint, never anywhere else. Each has its own audience claim. Each is signed with RS256. They are not interchangeable.
"Which token does the backend trust?"
The access token, not the ID token. The ID token's audience is the SPA's client ID — it's meant for the client. The access token's audience is the API identifier. Sending the wrong one is one of the most common mistakes I've seen in OAuth integrations.

Q&A · ~10 minutes

Anticipated questions

Ten questions from the architecture doc, plus the curveballs requirement #11 warns about. Each answer is 1–2 sentences — say it, then stop. Resist the urge to keep talking.

Why Authorization Code + PKCE, not Implicit?
Implicit puts tokens in the URL — they end up in browser history, server logs, and referer headers. Auth Code + PKCE keeps them in the response body of the token endpoint, and PKCE replaces the client secret with a one-time crypto challenge generated per login. It's the current OAuth 2.1 and IETF best practice for browser apps.
Why a separate Machine-to-Machine application?
Credential isolation — the SPA can't safely hold a secret, but the backend can. Different OAuth grant — SPA uses Authorization Code with PKCE, M2M uses Client Credentials, and Auth0 doesn't allow mixing grants in one app record. Least privilege — the M2M's Management API scopes are powerful and should never touch a browser. And audit clarity — every Auth0 log is tagged by which app made the call.
What about refresh tokens? How do you handle expiry?
Refresh tokens with rotation are enabled. Each refresh issues a new refresh token and revokes the previous one — if Auth0 ever sees the old one reused, it's a breach signal and the entire token family is revoked. The SPA uses refresh tokens over iframe silent renew because third-party cookies are increasingly blocked in modern browsers, and refresh tokens are back-channel POSTs that work everywhere.
Front channel vs back channel — what's the distinction?
Front channel is anything that travels through browser redirects — visible in URLs. The authorization request to /authorize goes through the front channel. Back channel is server-to-server HTTP. The token exchange at /oauth/token, JWKS fetches for signature verification, and all Management API calls go through the back channel. We use the front channel only for the initial redirect; all token traffic is back channel so tokens never appear in the URL.
Why save orders to user metadata? Isn't that an anti-pattern?
In production it would be — and I'd push back on this requirement. The right place for orders is Pizza 42's own database, with maybe a thin summary mirrored to app_metadata if it needs to ride in tokens. I made the spec'd choice for the PoC but used app_metadata not user_metadata, because orders are system-owned and there's no SPA-self-edit scope for app_metadata. That at least makes the order data tamper-proof.
Why put order history in the ID token? That doesn't scale.
Honest answer — for a real Pizza 42 it shouldn't. Tokens travel in headers, they have 24-hour lifetimes, and they carry blast-radius risk if they leak. In production I'd put a summary in the token — order count, last order timestamp — and fetch the full list on demand via a read:orders scope. The requirement asked for the full list, so I delivered it, but I'd flag this on a real customer engagement.
How would you do progressive profiling?
A Post-Login Action that triggers an additional flow on logins after the first — using api.redirect.sendUserTo() to redirect to a Pizza 42 page that captures one more field — favorite pizza, dietary preferences, delivery zone — and writes it to user_metadata. Capture is gradual, signup stays frictionless, and marketing gets a richer profile over time.
What's the GenAI angle?
Two layers. First — the order history claim in the ID token is already a structured customer context that a reorder-recommendation chatbot can consume directly. Second — Auth0's emerging Auth for GenAI gives agent-identity and delegated tool-use, so a Pizza 42 agent can act on a user's behalf with scoped, auditable permissions. It's identity infrastructure for AI agents, not just human users.
What changes at 10 million users?
Orders move to Pizza 42's own database; app_metadata only carries a summary if it's used in tokens. A custom domain like auth.pizza42.com for branding and first-party cookies. Refresh token rotation with shorter lifetimes. Adaptive MFA on higher-risk actions. Log streams piped to your SIEM. Account linking for users who arrive via multiple IdPs. And progressive profiling instead of one big signup form.
How do you rotate the M2M client secret?
Auth0 supports multiple credentials per M2M app. Generate the new credential, deploy it to env vars, shift traffic, then revoke the old one — zero downtime. For higher assurance you can switch the M2M from shared-secret authentication to private-key JWT, which is essentially mTLS-style auth and removes the shared-secret leak risk entirely.
What if Auth0 is down?
JWT validation keeps working as long as the backend has cached the JWKS. The SPA gracefully degrades to read-only or cached views for already-authenticated users. New logins are blocked, and there's no realistic mitigation for that without a multi-IdP fallback — which is a bigger architectural change with its own cost. The right answer for most customers is to lean on Auth0's 99.99% SLA and design for graceful degradation, not multi-IdP.
Why a live Mgmt API check for email_verified instead of trusting the token?
The token's email_verified is a snapshot from login time. A user who verifies mid-session would be blocked until they re-authenticated. The live lookup costs one extra HTTP call — which the backend is making anyway to save the order — and gives always-current state. We pair that backend authority with a focus-listener silent renewal so the UI catches up without a manual refresh.
How do you handle account linking — same person, two IdPs?
Account linking is enabled on the database connection. If a user signs up with email and later logs in with Google using the same email, Auth0 links the two identities into one user record. The order history follows the canonical user, not the auth method.
Pizza 42 wants B2B — corporate ordering for offices. How would you extend this?
Auth0 Organizations. Each corporate customer becomes an organization with its own branded login page, its own SSO connection (SAML or OIDC to their IdP), and its own roles. The same SPA serves both consumer and B2B modes — organization claim in the token tells the backend which tenant context to apply. Adds about a day's work on top of the existing architecture.
