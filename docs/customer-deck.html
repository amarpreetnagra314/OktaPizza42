Pizza42 + Auth04

OAuth 2.0 & OIDC Strategy

Every flow choice is intentional. Aligned with current IETF OAuth 2.0 Security BCP.

🔑 Authorization Code + PKCE SPA

Used by the React SPA. PKCE prevents authorization-code interception in browsers — the modern standard for public clients.

👤 OIDC for Identity User claims

ID token carries who the user is — name, email, verified status — for personalized UI.

🛠️ OAuth 2.0 for API Authz

Access token carries what the user is allowed to do. Backend validates and enforces scopes.

🤖 Client Credentials M2M

Server-to-server credentials never leave the backend. Used only for privileged Management API operations.

❌ Implicit Flow and ROPC explicitly rejected — known vulnerabilities, deprecated by IETF.
Pizza42 + Auth05

End-to-End Token Flow

Every step enforces a trust boundary. Privileged operations stay server-side.

1️⃣

User Authenticates

React SPA redirects to Universal Login. User chooses email/password, Google, or Passkey. Auth0 handles credentials.

2️⃣

Auth0 Issues Tokens

ID token to the SPA (identity), Access token to the backend (authorization). Both signed with RS256.

3️⃣

Backend Validates & Updates

API verifies JWT and scope. M2M token used to write order to Auth0 user profile via Management API.

The SPA never directly accesses the Management API — all privileged operations flow through the validated backend, ensuring credentials and sensitive operations stay server-side.
Pizza42 + Auth06

Security Controls

Defense in depth — multiple independent gates, each with one job.

📜

JWT Validation

RS256 asymmetric signing verified via JWKS endpoint — no shared secrets.

🎯

Scope Enforcement

Backend explicitly checks create:orders on every request — authentication + authorization, not just one.

📧

Email Verification

Server-side gate via Management API — always current, prevents unverified orders even if UI is bypassed.

🔐

Secrets Isolation

M2M client secrets never exposed to the browser — stored only in server environment variables.

🔄

Token Rotation

Refresh-token rotation with reuse detection — leaked tokens invalidate the entire family.

🛡️

Tamper-proof Data

Orders stored in app_metadata — no user-self-service scope exists, structurally unwritable from the browser.

Pizza42 + Auth07

Scopes vs. RBAC

Start lean, scale to roles when the business needs it — no refactoring required.

📌

Today: Lightweight Scopes

PoC uses fine-grained scopes like create:orders — simple, explicit, enforced in backend middleware.

Ideal for single-persona applications where role complexity is low.

📈

Next: RBAC at Scale

As Pizza42 grows to support customer, staff, and admin personas, Auth0 RBAC turns on with a tenant-level toggle.

No frontend refactoring required — backend already checks both scope and permissions claims.

✅ Auth0 RBAC can be enabled at the tenant level with no frontend refactoring required.
